# 核心概念与架构

理解 Claude Code 的底层原理是高效使用的关键。本章从架构、工作流、上下文管理等维度深入剖析。

## 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户 (User)                               │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                    CLI 交互层 (Node.js)                          │
│  - 命令解析 (/commands, /model, /clear)                          │
│  - 会话管理 (历史、压缩、恢复)                                    │
│  - 输出渲染 (Markdown、diff、进度条)                              │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Agent 引擎                                    │
│  - 任务规划 (分解为子任务)                                        │
│  - 工具调用 (Read、Write、Bash、Grep 等)                         │
│  - 上下文管理 (自动压缩、优先级排序)                              │
│  - 子代理调度 (调用其他 Agent)                                    │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Claude API (Anthropic)                        │
│  - 消息处理 (user/assistant 交替)                                │
│  - 工具定义 (functions calling)                                 │
│  - 响应生成 (流式输出)                                           │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                    本地环境 (沙箱)                                │
│  - 文件系统 (Read/Write/Edit)                                    │
│  - Git 操作 (status、commit、push)                               │
│  - Shell 命令 (Bash)                                            │
│  - MCP 工具 (外部服务)                                           │
└─────────────────────────────────────────────────────────────────┘
```

## 工作模式对比

| 模式 | 特点 | 适用场景 | 示例 |
|------|------|----------|------|
| **交互模式** | 持续对话，保留上下文 | 复杂任务、多轮迭代 | `claude` 启动后连续提问 |
| **单次执行模式** | 执行单个任务后退出 | 脚本集成、CI/CD | `claude -p "解释这个函数"` |
| **Agent 模式** | 自主决策、多步操作 | 大型重构、自动化流程 | 自动分析 → 规划 → 执行 → 验证 |
| **子代理模式** | 委托特定任务给专业 Agent | 代码审查、安全扫描 | `code-reviewer` 代理独立运行 |

## 上下文管理

### 上下文组成

Claude Code 的上下文窗口为 **100 万 token**，组成如下：

```
┌──────────────────────────────────────────────┐
│  系统提示词 (System Prompt)         ~5K      │
├──────────────────────────────────────────────┤
│  CLAUDE.md (项目配置)              ~2K      │
├──────────────────────────────────────────────┤
│  Rules (规则文件)                  ~10K     │
├──────────────────────────────────────────────┤
│  Memory (持久化记忆)               ~5K      │
├──────────────────────────────────────────────┤
│  工具定义 (Tools/MCP)              ~20K     │
├──────────────────────────────────────────────┤
│  会话历史 (对话+工具结果)          ~958K    │
└──────────────────────────────────────────────┘
```

### 自动压缩机制

当上下文接近上限时（默认 95%），Claude Code 会：

1. **识别关键信息**：保留 CLAUDE.md、Rules、最近的对话
2. **压缩旧内容**：将早期对话提炼为摘要
3. **丢弃冗余**：移除重复的工具调用结果

**配置压缩阈值**：
```json
// ~/.claude/settings.json
{
  "env": {
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50"  // 50% 时主动压缩
  }
}
```

### 手动压缩 vs 自动压缩

| 操作 | 触发方式 | 效果 | 适用场景 |
|------|----------|------|----------|
| `/clear` | 手动 | 清空所有历史，完全重置 | 开始全新任务 |
| `/compact` | 手动 | 保留关键上下文，压缩细节 | 任务进行到里程碑 |
| 自动压缩 | 阈值触发 | 同上，但不可控 | 防止超限 |

**最佳实践**：在逻辑断点手动执行 `/compact`，而非依赖自动压缩。

## 工具调用机制

Claude Code 通过工具与外部世界交互。每个工具调用是一个标准化的 JSON 请求。

### 工具调用流程

```
1. 用户发送请求
         │
         ▼
2. Claude 分析 → 决策需要调用工具
         │
         ▼
3. 返回 tool_use 消息
   {
     "type": "tool_use",
     "name": "Read",
     "input": { "file_path": "/src/index.ts" }
   }
         │
         ▼
4. 系统执行工具（沙箱环境）
         │
         ▼
5. 返回 tool_result 消息
   {
     "type": "tool_result",
     "content": "文件内容..."
   }
         │
         ▼
6. Claude 分析结果 → 继续或最终响应
```

### 内置工具详解

| 工具 | 功能 | 关键参数 | 安全限制 |
|------|------|----------|----------|
| **Read** | 读取文件 | `file_path`, `offset`, `limit` | 不能读取二进制、`.env` |
| **Write** | 创建/覆盖文件 | `file_path`, `content` | 需要确认 |
| **Edit** | 精确替换字符串 | `file_path`, `old_string`, `new_string` | 需要确认 |
| **Bash** | 执行 shell 命令 | `command`, `timeout` | 危险命令需确认 |
| **Glob** | 模式匹配查找文件 | `pattern`, `path` | 只读 |
| **Grep** | 内容搜索（ripgrep） | `pattern`, `path`, `-i`, `-n` | 只读 |
| **TodoWrite** | 管理任务列表 | `todos` (数组) | 无 |
| **AskUserQuestion** | 向用户提问 | `questions`, `answers` | 无 |

### 并行工具调用

Claude Code 支持**在同一轮响应中调用多个独立工具**，大幅提升效率。

**示例**：需要同时读取 3 个文件
```json
[
  {"type": "tool_use", "name": "Read", "input": {"file_path": "/src/a.ts"}},
  {"type": "tool_use", "name": "Read", "input": {"file_path": "/src/b.ts"}},
  {"type": "tool_use", "name": "Read", "input": {"file_path": "/src/c.ts"}}
]
```

**依赖场景**：必须先读取再编辑，则顺序调用。

## Agent 引擎详解

### Agent 决策循环

```
┌─────────────────────────────────────────────────────────────┐
│                    收到用户任务                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 1: 分析 (Analyze)                                      │
│  - 理解意图                                                  │
│  - 识别所需工具                                              │
│  - 评估复杂度                                                │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 2: 规划 (Plan)                                         │
│  - 分解为子任务                                              │
│  - 确定执行顺序                                              │
│  - 标记依赖关系                                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 3: 执行 (Execute)                                      │
│  - 调用工具                                                  │
│  - 收集结果                                                  │
│  - 处理错误                                                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 4: 验证 (Verify)                                       │
│  - 检查结果是否符合预期                                       │
│  - 运行测试/类型检查                                          │
│  - 必要时回退或调整                                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    是否还有未完成的子任务？
                    ├── 是 → 回到阶段 3
                    └── 否 → 输出最终结果
```

### 子代理 (Subagent) 机制

当任务过于复杂或需要专门知识时，主 Agent 可以**委托**给子代理。

**子代理的特点**：
- 独立的上下文窗口（不污染主对话）
- 专门的系统提示词（如代码审查专家）
- 受限的工具集（只能使用特定工具）
- 返回结果后销毁

**配置子代理**：
```markdown
# ~/.claude/agents/code-reviewer.md
---
name: code-reviewer
description: 专业的代码审查专家
tools: ["Read", "Grep", "Glob"]
model: haiku
---

你是代码审查专家，专注以下方面：
1. 类型安全（TypeScript）
2. 错误处理
3. 性能问题
4. 安全漏洞
```

## MCP 协议集成

MCP (Model Context Protocol) 是连接 AI 与外部工具的标准协议。

### 架构对比

**无 MCP**：
```
Claude Code → 内置工具 (Read, Write, Bash) → 本地环境
```

**有 MCP**：
```
Claude Code → MCP Client → MCP Server → 外部服务
                              ├── GitHub API
                              ├── PostgreSQL
                              ├── Slack
                              └── 任意自定义服务
```

### MCP 工具调用流程

```
1. MCP Server 启动时注册工具列表
         │
         ▼
2. Claude Code 接收工具定义（如 github_create_issue）
         │
         ▼
3. Claude 决定调用 github_create_issue
         │
         ▼
4. 通过 MCP 协议发送 JSON-RPC 请求
         │
         ▼
5. MCP Server 执行实际 API 调用
         │
         ▼
6. 返回结果给 Claude
```

## Rules 规则引擎

Rules 是始终生效的指令，告诉 Claude 如何行为。

### 规则优先级

```
项目级 Rules (.claude/rules/*.md)     ← 最高优先级
         │
         ▼
用户级 Rules (~/.claude/rules/*.md)
         │
         ▼
CLAUDE.md 中的规则
         │
         ▼
默认行为                           ← 最低优先级
```

### 规则示例

```markdown
# .claude/rules/testing.md
---
description: 测试规范
globs: ["**/*.test.ts", "**/*.spec.ts"]
---

- 每个测试文件必须对应一个源文件
- 使用 describe/it 结构
- 测试命名: "should <预期行为> when <条件>"
- 覆盖率目标 > 80%
```

## Hooks 钩子系统

Hooks 在特定事件发生时**自动执行脚本**。

### 支持的事件

| 事件 | 触发时机 | 典型用途 |
|------|----------|----------|
| `PreToolUse` | 工具调用前 | 权限检查、成本预警 |
| `PostToolUse` | 工具调用后 | 自动格式化、日志记录 |
| `Stop` | Agent 停止时 | 会话总结 |
| `SessionStart` | 会话开始时 | 加载记忆 |
| `SessionEnd` | 会话结束时 | 保存学习成果 |
| `PreCompact` | 压缩前 | 保存关键状态 |

### Hook 配置示例

```json
// ~/.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "tool == \"Bash\" && tool_input.command matches \"rm -rf\"",
        "hooks": [
          {
            "type": "command",
            "command": "echo '危险命令被拦截' && exit 1"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "tool == \"Edit\" && tool_input.file_path matches \"\\.ts$\"",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write ${tool_input.file_path}"
          }
        ]
      }
    ]
  }
}
```

## Memory 记忆系统

### 记忆类型

| 类型 | 存储位置 | 生命周期 | 用途 |
|------|----------|----------|------|
| 会话记忆 | 内存 | 当前会话 | 临时上下文 |
| 项目记忆 | `.claude/memory/` | 项目生命周期 | 项目特定信息 |
| 用户记忆 | `~/.claude/memory/` | 永久 | 个人偏好 |

### 记忆自动保存

通过 Hooks 实现跨会话记忆：

```javascript
// hooks/session-end.js
// 在会话结束时提取模式并保存
const patterns = extractCodePatterns(sessionHistory);
appendToMemoryFile(patterns);
```

## 性能与成本

### Token 消耗估算

| 操作 | 平均 Token | 成本 (Sonnet) | 成本 (Opus) |
|------|-----------|--------------|-------------|
| 读取 100 行代码 | ~500 | $0.001 | $0.003 |
| 编辑 50 行代码 | ~1000 | $0.002 | $0.006 |
| 运行一次测试 | ~2000 | $0.004 | $0.012 |
| 复杂重构任务 | 10000+ | $0.02 | $0.06 |

### 优化策略

1. **默认使用 Sonnet**：覆盖 80% 任务，成本降低 60%
2. **限制思考 Token**：`MAX_THINKING_TOKENS=10000`
3. **主动压缩**：里程碑处手动 `/compact`
4. **禁用无用 MCP**：每个 MCP 工具定义消耗 ~2K token

```json
// 优化配置
{
  "model": "sonnet",
  "env": {
    "MAX_THINKING_TOKENS": "10000",
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50",
    "CLAUDE_CODE_SUBAGENT_MODEL": "haiku"
  }
}
```

## 总结

| 概念 | 一句话解释 | 关键作用 |
|------|-----------|----------|
| Agent | 自主决策的 AI 实例 | 执行复杂任务 |
| Tool | 与外部环境交互的能力 | 读文件、执行命令 |
| MCP | 连接外部服务的标准协议 | 扩展能力边界 |
| Rule | 始终生效的指令 | 确保一致性 |
| Hook | 事件触发的自动化脚本 | 工作流集成 |
| Memory | 跨会话持久化信息 | 积累上下文 |
